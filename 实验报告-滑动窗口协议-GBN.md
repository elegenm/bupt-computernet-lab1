## 3 协议设计

### 3.1 协议分层结构及各层功能

本实验的协议软件按照典型的分层思想组织：

- 网络层：负责产生固定长度为 256 字节的分组，并向数据链路层提供 `get_packet()` / `put_packet()` 接口
- 数据链路层：本实验需要自行设计与实现的核心部分，负责成帧、确认、重传、差错检测和流量控制
- 物理层：由实验平台提供，负责按照给定带宽、传播时延和误码率发送字节流

其中，`protocol.h` 已经定义好了协议层之间的接口。程序通过 `wait_for_event()` 接收平台事件，再根据不同事件调用对应逻辑，因此本实验本质上是一个事件驱动的数据链路层协议实现。

### 3.2 程序总体结构

整个工程中，`protocol.c` 负责仿真环境，`datalink_gobackn.c` 负责协议主体。其总体结构可以概括为：

1. 平台初始化网络层、物理层和定时器环境。
2. 数据链路层进入主事件循环。
3. 当网络层就绪时，取出分组并发送新数据帧。
4. 当收到帧时，先做 CRC 校验，再根据帧类型分别处理 ACK 或 DATA。
5. 当数据帧超时后，从发送窗口左边界开始整窗重传。
6. 每轮事件结束后，根据窗口是否已满、物理层是否可发来决定是否重新打开网络层。

### 3.3 Go-Back-N 协议设计原理

Go-Back-N 的基本思想是：发送方可以在收到确认前连续发送多个帧，但接收方只接收“按序到达”的下一帧。如果某一帧出错或丢失，后续帧即使到达也会被丢弃；发送方在超时后从最早未确认帧开始，把整个未确认窗口重新发送。

本实现的关键状态变量包括：

- `ack_expected`：发送窗口左边界，表示最早尚未被确认的序号
- `next_frame_to_send`：下一个新帧可使用的序号
- `frame_expected`：接收方当前期待收到的序号
- `nbuffered`：窗口中已经发送但尚未确认的帧数

由此可知，发送窗口可以抽象为循环区间 `[ack_expected, next_frame_to_send)`。接收窗口在 Go-Back-N 中退化为“只接收一个期待序号”，即只有 `seq == frame_expected` 的数据帧才会被提交给网络层。

### 3.4 帧结构定义

本实验中使用的数据链路层帧类型定义在 `datalink.h` 中，包括：

- `FRAME_DATA`
- `FRAME_ACK`
- `FRAME_NAK`

当前 Go-Back-N 实现实际使用的是 `DATA` 和 `ACK` 两类帧，未启用 `NAK`。帧结构定义如下：

```c
struct FRAME {
    unsigned char kind;
    unsigned char ack;
    unsigned char seq;
    unsigned char data[PKT_LEN];
    unsigned int  padding;
};
```

对应的有效长度为：

- `DATA` 帧：`1(kind) + 1(ack) + 1(seq) + 256(data) + 4(CRC) = 263 B`
- `ACK` 帧：`1(kind) + 1(ack) + 4(CRC) = 6 B`

其中 `padding` 仅用于给 CRC 写入预留空间，不参与链路层协议语义。

### 3.5 实现可靠通信和误码控制

可靠通信由以下机制共同保证：

- 每发送一个数据帧，立即为其启动重传定时器
- 接收方只接收按序帧，保证提交给网络层的数据不乱序、不重复
- 发送方使用累计确认，收到 ACK 后可连续滑动发送窗口
- 一旦任意未确认帧超时，从窗口左边界开始整窗重传
- 每个待发送帧在 `out_buf` 中保留副本，确保重传内容与初次发送一致
- 所有帧在发送前计算 CRC32，接收时重新校验 CRC，校验失败的帧直接丢弃

需要说明的是，本实现属于较为纯粹的 Go-Back-N 版本：

- 不缓存乱序帧
- 不使用 NAK
- 不使用独立 ACK 超时器
- 收到数据后立即返回 ACK

这种设计逻辑清晰，便于实现和调试，但在误码率上升时会因为整窗回退而损失较多带宽利用率。

## 4 软件设计

### 4.1 数据结构

本实现中最核心的数据结构和常量如下：

```c
#define MAX_SEQ      7
#define WINDOW_SIZE  MAX_SEQ
#define DATA_TIMER   2000
```

其意义分别为：

- `MAX_SEQ = 7`：序号空间为 `0~7`，共 8 个序号
- `WINDOW_SIZE = 7`：发送窗口大小为 7，满足 Go-Back-N 中“窗口大小严格小于序号空间大小”的要求
- `DATA_TIMER = 2000`：单个数据帧的重传超时时间设为 2000 ms

发送缓存定义为：

```c
static unsigned char out_buf[MAX_SEQ + 1][PKT_LEN];
```

这样每个序号都对应一个缓存槽位，便于协议在发生超时时按原内容重发。

### 4.2 模块结构和算法思路

#### 4.2.1 `inc()` 函数

该函数用于序号自增，并自动处理回绕：

```c
*num = (*num + 1) % (MAX_SEQ + 1);
```

因为序号空间是环形的，`7` 的下一个序号不是 `8`，而是重新回到 `0`。

#### 4.2.2 `between()` 函数

`between(a, b, c)` 用于判断 `b` 是否落在循环区间 `[a, c)` 内，是滑动窗口处理中最关键的辅助函数之一。由于序号会回绕，不能直接使用普通整数区间判断，因此该函数专门处理如 `[6, 2)` 这种跨越回绕点的窗口。

该函数主要用于判断收到的 ACK 是否仍落在当前发送窗口中，从而决定窗口能否继续向前滑动。

#### 4.2.3 `put_frame()` 函数

`put_frame()` 的作用是：

1. 对帧头和数据部分计算 CRC32
2. 将计算结果追加到帧尾
3. 调用平台提供的 `send_frame()` 把帧发往物理层
4. 将 `phl_ready` 置为 0，等待下一次 `PHYSICAL_LAYER_READY`

该函数实现了链路层帧真正“出站”前的最后一步封装。

#### 4.2.4 `send_data_frame()` 函数

该函数负责按指定序号构造一个 `DATA` 帧并发送：

- `seq` 字段填入当前帧自己的序号
- `ack` 字段填入“最近一个按序正确收到的帧号”
- `data` 字段从对应缓存槽 `out_buf[frame_nr]` 中复制
- 发送后启动该序号对应的数据定时器

这说明当前实现虽然协议行为是 Go-Back-N，但在定时器管理上采用了“每个已发送帧一个定时器”的实现方式，代码较直观，便于调试。

#### 4.2.5 `send_ack_frame()` 函数

当接收端收到一个数据帧时，无论该帧是否按序，都会发送一个 ACK，ACK 中携带的是当前最新的累计确认号。这样发送方能够及时获知“对方已经连续按序收到哪里了”。

#### 4.2.6 `resend_window()` 函数

该函数体现了 Go-Back-N 的核心重传策略。只要当前窗口中任意一个帧超时，就从 `ack_expected` 开始，把整个未确认窗口中所有帧按顺序重新发送一遍。

例如，若未确认帧为 `3,4,5`，则无论是 `4` 超时还是 `5` 超时，最终都会从 `3` 开始重传 `3,4,5`。

### 4.3 函数调用和程序流程

主流程集中在 `main()` 的事件循环中，其逻辑可以概括为：

1. 调用 `protocol_init()` 初始化协议平台。
2. 先关闭网络层，避免窗口未初始化完成时过早取包。
3. 反复等待平台事件 `wait_for_event(&arg)`。
4. 根据事件类型执行不同动作。

详细事件处理过程如下：

#### 4.3.1 `NETWORK_LAYER_READY`

- 从网络层取出一个分组
- 放入 `out_buf[next_frame_to_send]`
- `nbuffered++`
- 发送该新帧
- `next_frame_to_send` 自增

这一过程体现了滑动窗口的“流水线发送”能力：当前面帧尚未确认时，后续帧仍可继续进入链路。

#### 4.3.2 `PHYSICAL_LAYER_READY`

- 将 `phl_ready` 置为 1

该事件只表示物理层当前允许再发送一帧，真正是否允许网络层继续交付新分组，要结合窗口是否已满统一判断。

#### 4.3.3 `FRAME_RECEIVED`

收到帧后，处理流程为：

1. 读取帧内容
2. 检查长度和 CRC
3. 若为损坏帧，则直接丢弃
4. 若为 `ACK` 帧，则根据累计确认滑动发送窗口
5. 若为 `DATA` 帧，则判断其序号是否等于 `frame_expected`
6. 若按序到达，则交付给网络层并推进 `frame_expected`
7. 无论是否按序，都返回一个 ACK 表示当前连续接收进度

其中窗口滑动使用如下思路：

- 只要收到的 ACK 覆盖了当前窗口左边界
- 就连续停止对应定时器
- 让 `ack_expected` 不断向右推进

因此，一个 ACK 有可能一次性确认多个窗口中的帧。

#### 4.3.4 `DATA_TIMEOUT`

当某个数据定时器到期时：

- 打印超时事件
- 调用 `resend_window()`
- 从最早未确认帧开始重传整个窗口

这一步是 Go-Back-N 与选择重传最本质的区别之一。

#### 4.3.5 网络层启停控制

每次事件处理结束后，程序都会统一判断是否重新打开网络层：

- 当 `nbuffered < WINDOW_SIZE` 且 `phl_ready == 1` 时，调用 `enable_network_layer()`
- 否则调用 `disable_network_layer()`

这相当于把发送窗口本身作为流量控制手段，从而避免发送方在链路还未准备好时无限制地继续塞入新数据。

## 5 关键参数的设计

### 5.1 序列号大小 `MAX_SEQ`

当前实现中取 `MAX_SEQ = 7`，因此序号空间大小为 8，窗口大小为 7。这样的选择有两层考虑：

- 从协议正确性上看，Go-Back-N 必须满足 `WINDOW_SIZE < MAX_SEQ + 1`，否则序号回绕后旧帧和新帧会混淆
- 从实现复杂度上看，8 个序号便于调试和观察窗口滑动过程，适合作为课程实验的基础实现

根据当前信道条件，数据帧传输时间约为：

```text
263 B × 8 / 8000 bps ≈ 263 ms
```

若按经典满流水条件估算，窗口大小只要达到 4 左右即可基本填满链路，因此窗口大小取 7 具有一定冗余，能够保证在低误码环境下稳定接近信道上限。

### 5.2 重传定时时延 `DATA_TIMER`

本实现取 `DATA_TIMER = 2000 ms`。粗略估算一次成功往返时间：

- 单向传播时延：`270 ms`
- 数据帧发送时间：约 `263 ms`
- ACK 帧发送时间：约 `6 ms`
- 反向传播时延：`270 ms`

因此单次往返时间大致在 `800 ms` 左右，再考虑平台调度粒度和噪声环境下的抖动，`2000 ms` 能够避免在正常情况下过早误判超时，同时又不至于把真正丢帧的恢复时间拖得过长。

### 5.3 ACK 定时策略

与学长报告中基于选择重传的 `ACK_TIMEOUT` 设计不同，当前 Go-Back-N 版本没有启用独立 ACK 定时器。理由如下：

- 该实现收到 DATA 后立即发送 ACK，不依赖延迟确认
- 平台虽然提供了 `ACK_TIMEOUT` 事件接口，但当前协议未使用
- 这样的设计降低了实现复杂度，也避免了 ACK 定时策略与 DATA 超时策略相互干扰

其代价是：当反向数据流较少时，纯 ACK 帧的发送会更频繁，理论上会带来少量控制开销。

## 6 实验结果分析

### 6.1 实验结果概述

为了使结果与当前代码一致，我将 `Lab1-Windows-VS2017/datalink_gobackn.c` 在 `Lab1-linux` 目录下独立编译为 `gobackn_current`，并分别启动 A、B 两个站点进行短时洪泛测试。测试表明：

- 在无误码信道中，协议可以稳定运行，信道利用率接近 95%
- 在默认误码率 `10^-5` 下，协议仍能保持可靠通信，但一旦出现差错，吞吐量会阶段性下降
- 在较高误码率 `10^-4` 下，协议性能显著下降，体现出 Go-Back-N 对误码较为敏感的特征

### 6.2 协议参数选择

本次测试使用的核心参数如下：

| 参数 | 取值 | 说明 |
| --- | --- | --- |
| `MAX_SEQ` | 7 | 序号空间为 8 |
| `WINDOW_SIZE` | 7 | 满足 `W < N` 的 Go-Back-N 约束 |
| `DATA_TIMER` | 2000 ms | 数据帧重传超时 |
| `PKT_LEN` | 256 B | 平台固定分组长度 |
| `CHAN_BPS` | 8000 bps | 平台固定带宽 |
| `CHAN_DELAY` | 270 ms | 平台固定单向传播时延 |

### 6.3 理论性能分析

若只从帧格式出发，不考虑启动/停止损耗和独立 ACK 的影响，数据帧的有效载荷占比为：

```text
η_frame = 256 / 263 ≈ 97.34%
```

这给出了当前帧结构在理想情况下的上限。实际运行时，性能还会受到以下因素影响：

- ACK 帧也要占用反向链路
- 事件驱动模型存在调度粒度
- 窗口起始填充阶段不能立即满流水
- 一旦出现误码，Go-Back-N 会从最早未确认帧开始整窗重传

因此，实测利用率略低于 97.34% 是正常现象。在误码率较高时，由于乱序帧会被丢弃，同一个数据分组可能被反复重发，利用率会明显下降。

| `-f -t 20` | B | 54 | 6244 bps | 78.04% | 1 |
| `-f -b 1e-4 -t 12` | A | 9 | 2320 bps | 29.00% | 6 |
| `-f -b 1e-4 -t 12` | B | 17 | 3446 bps | 43.08% | 5 |

按双向平均值统计：

| 场景 | 平均吞吐率 | 平均利用率 | 现象 |
| --- | --- | --- | --- |
| 无误码 | 7567 bps | 94.59% | 接近帧结构上限，窗口可以持续满流水 |
| 默认误码率 `10^-5` | 6262 bps | 78.27% | 少量差错会触发整窗回退，吞吐率明显波动 |
| 高误码率 `10^-4` | 2883 bps | 36.04% | 频繁重传，Go-Back-N 性能显著下降 |

从上述数据可以得到两个结论：

1. 在低误码或无误码环境下，Go-Back-N 能较充分利用给定卫星信道的带宽。
2. 在误码率升高时，整窗重传会使后续原本已到达但乱序的数据全部失效，从而造成明显的带宽浪费。

这与 Go-Back-N 协议的理论特征完全一致，也说明当前实现逻辑基本正确。

## 7 研究探索问题

### 7.1 CRC 校验能力

课程平台提供了 `crc32.c` 用于帧的差错检测。CRC32 的优点在于：

- 能够检测所有单比特错误
- 能够检测大多数突发错误
- 冗余开销相对可控，适合工程实现

在本实验中，CRC 的作用主要是“检测帧是否损坏”，而不是纠错。也就是说，若接收端发现 CRC 错误，就直接丢弃该帧，并依靠上层 ARQ 机制通过超时重传完成恢复。

### 7.2 CRC 校验和的计算方法

`crc32.c` 采用查表方式完成 CRC32 计算。与逐位多项式除法相比，查表法把大量重复计算预先压缩成固定查找表，从而在运行时只需按字节更新 CRC 寄存器，效率更高，也更适合在协议主循环中频繁调用。

在本实验中，CRC 的使用方式为：

- 发送前：对帧头和数据部分计算 CRC，并追加到帧尾
- 接收后：对整个帧重新计算 CRC，结果为 0 视为通过

这种设计使差错检测逻辑集中、清晰，也降低了数据链路层代码的复杂度。

### 7.3 程序设计方面的问题

本次实现中有几个值得注意的设计点：

- 当前代码虽然属于 Go-Back-N，但使用了“每个已发送帧一个定时器”的实现方式，而经典教材中的 GBN 常常只给最早未确认帧设置一个总定时器。当前做法的优点是直观，便于调试；缺点是计时器管理略繁琐。
- 当前实现没有使用 `NAK`。这使协议逻辑更简洁，但也意味着接收端只能被动等待发送方超时后重传，不能像带 NAK 的 SR 那样更快地提示某一缺失帧。
- 当前实现不缓存乱序帧，因此性能高度依赖信道质量。一旦某一帧出错，后续正确到达的帧也会被浪费，这正是 GBN 的代价。

### 7.4 软件测试方面的问题

本实验的一个难点在于：短时测试很容易受到随机误码波动影响。即使参数完全相同，A、B 两端在有限测试时长内看到的误码次数也可能不同，因此最终吞吐率和利用率会有偏差。

因此在分析实验结果时，需要注意：

- 用多组不同条件测试，而不是只看单次结果
- 把日志中的吞吐率变化与错误次数一起分析
- 区分“协议实现错误”与“随机误码导致的性能波动”

当前报告中采用了三组对比测试，已经足以体现当前 GBN 实现对误码率变化的敏感性。

### 7.5 对等协议实体之间的流量控制

在本实现中，流量控制主要依靠发送窗口完成。只要 `nbuffered` 达到 `WINDOW_SIZE`，网络层就会被关闭，不再继续向下交付新分组。这样可以避免发送方发送速度过快，导致物理层或接收方被未确认帧淹没。

从实现角度看，窗口在这里同时承担了两类职责：

- 可靠传输中的未确认帧管理
- 对等实体之间的发送速率约束

因此，窗口机制本身就构成了一个简化但有效的流量控制策略。

### 7.6 与标准协议的对比

课程实验中的协议是一个教学性质很强的简化模型，与真实链路层协议相比仍有明显差距：

- 当前实现是面向实验平台的成帧协议，不涉及位填充、链路建立和链路拆除
- 未实现更复杂的异常恢复、链路探测和长时间空闲处理
- 未实现选择重传、乱序缓存和更细粒度的接收控制

与 HDLC/LAPB 等成熟协议相比，本实现保留了“编号、确认、定时器、窗口、CRC”这些核心思想，但省略了许多工程细节。正因为如此，它非常适合用于理解数据链路层协议的基本机制，却不能直接等价看作现实通信系统中的完整协议栈。

## 8 实验总结和心得体会

### 8.1 实验总结

本次实验完成了一个可运行的 Go-Back-N 滑动窗口协议，实现了以下关键能力：

- 数据帧封装与发送
- 累计确认
- 发送窗口滑动
- 定时器超时重传
- CRC32 差错检测
- 基于窗口的流量控制

从实验结果看，该实现能够在默认误码环境下保持可靠通信，并在无误码场景中接近信道上限运行；在高误码环境下，吞吐率会明显下降，这与 Go-Back-N 的协议特征一致，说明实验实现与理论分析相互吻合。

### 8.2 心得体会

通过本实验，可以明显感受到“协议能运行”和“协议运行得好”是两个不同层次的问题。前者要求把发送、确认、重传和校验等基本机制实现正确；后者则要求进一步考虑窗口大小、超时设置、误码恢复效率和测试方法等细节。

相比只在课堂上理解停等协议和滑动窗口原理，自己动手实现以后，会更清楚以下问题：

- 为什么 ACK 需要累计确认
- 为什么序号空间和窗口大小必须满足约束
- 为什么误码率一升高，GBN 的效率就会迅速下降
- 为什么“接收端是否缓存乱序帧”会直接决定协议性能上限

这类体会仅靠理论阅读很难建立，必须通过编码、调试和看日志才能真正形成完整认识。
